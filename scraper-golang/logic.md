# Логика скрапера (по `scraper-python`)

Документ описывает **пайплайн и правила поведения**, как они заложены в Python-версии (`mpscraper`, точка входа CLI, режим watch). Это не спецификация API и не привязка к языку — только логика процессов.

---

## 1. Точка входа и окружение

1. Регистрируется обработка сигнала завершения, чтобы корректно прерывать долгий прогон.
2. Настраивается логирование; уровень шума от HTTP-клиентов при необходимости понижается.
3. Читаются аргументы командной строки и при необходимости переопределяются переменными окружения (лимиты, таймауты, headless, прокси и т.д.).
4. Прокси (исторически Python): аргументы, JSON-файл или переменная окружения; для Go см. **раздел 9**.
5. Если запрошен не headless и нет дисплея — поднимается виртуальный дисплей; если не удалось — ошибка.
6. Проверяется наличие исполняемого Chromium и при указании — ChromeDriver.
7. Отдельный режим «отслеживание кликов» обходит основной пайплайн: открывается браузер для ручной отладки селекторов.

Основной режим требует модуль БД и отправки в Telegram: без них запуск с сохранением в БД считается невозможным.

---

## 2. Загрузка состояния из БД и политика пересканирования

1. Из БД загружаются уже известные объявления и множество их идентификаторов.
2. Если в таблице есть время последнего обхода объявления, для записей старше порога **в часах** идентификатор **убирается** из множества «уже видели» — так объявление снова может попасть в выборку при следующем проходе (пересканирование по возрасту записи в БД).

Итог: при скрапинге «новое» — это то, чего нет в текущем множестве известных ID с учётом правила пересканирования.

---

## 3. Стартовая проверка и получение списка родительских категорий

1. Перед основной работой выполняется **пробный** запуск браузера с перебором прокси по очереди (если прокси заданы).
2. Цель этапа — успешно открыть главную страницу маркетплейса и вызвать получение **родительских категорий**.
3. При ответе «запрещено» с прокси — пауза, закрытие браузера, следующий прокси; без прокси ошибка пробрасывается.
4. При сбоях, похожих на прокси (таймауты, ошибки туннеля и т.п.) — аналогично: пауза, смена прокси.
5. Если ни один прокси не подошёл — ошибка уровня «категории недоступны».
6. После успешного получения списка категорий **стартовый** экземпляр скрапера закрывается; дальше категории обрабатываются в основном цикле (см. ниже).

Родительские категории извлекаются с главной: сначала разбор HTML-разметки блока категорий; если пусто — запасной путь через глобальный объект конфигурации страницы (ссылки категорий с идентификаторами и путями).

---

## 4. Режим одного воркера (последовательно)

1. Снова перебор прокси при необходимости; для каждого варианта создаётся рабочий экземпляр скрапера с браузером.
2. Ведётся **оставшийся лимит** новых объявлений (если лимит задан); ноль означает «без жёсткого потолка по количеству новых».
3. Для каждой родительской категории по порядку вызывается получение списка объявлений с передачей:
   - лимита (сколько ещё можно набрать),
   - множества уже известных ID,
   - опционально — функции обратного вызова на каждую **пачку** (страницу) новых карточек,
   - порога **максимального возраста** объявления в часах (используется в сценариях вроде «только свежие»).
4. После каждого успешного подкаталога новые ID добавляются в общее множество; лимит уменьшается на фактически добавленные новые карточки.
5. Исключения:
   - **Устаревшая категория**: встретилось объявление старше порога — накопленный список сохраняется в исключении, категория пропускается, цикл идёт дальше.
   - **Ошибка списка / прерывание пользователем**: возвращается то, что успели собрать; при прерывании выставляется флаг остановки всего прохода.
   - **Прокси / 403**: при привязке к прокси 403 трактуется как сигнал сменить прокси; иначе — как общая ошибка доступа.
6. Параллельно с накоплением данных срабатывает **пакетное сохранение**: при накоплении не меньше порогового числа новых строк выполняется слияние с текущим датафреймом, удаление дубликатов и запись в БД; опционально пачка уходит в пакетную отправку в Telegram.
7. В конце полное сохранение оставшегося датафрейма в БД, если есть что сохранять.

---

## 5. Режим нескольких воркеров (параллельно)

1. Число воркеров ограничивается минимумом из «запрошенное» и «число категорий».
2. Список родительских категорий **раскладывается по кругу** по воркерам (первый воркер получает категории 0, N, 2N…, второй — 1, N+1…).
3. **Лимит новых объявлений** делится между воркерами с округлением вверх по воркеру (каждый крутит свой подлимит).
4. Каждый воркер — отдельный браузер; прокси-список для воркера **сдвигается** по индексу воркера, чтобы разные воркеры стартовали с разных прокси; при сбое — тот же механизм перебора следующих прокси, что и в однопоточном режиме.
5. Результаты со всех воркеров собираются; прогресс по пачкам уходит в **общую очередь**.
6. Отдельный поток-потребитель очереди: для каждой пачки — попытка пакетной отправки в Telegram (ошибки не валят процесс), накопление до порога — то же слияние, дедупликация и upsert в БД.
7. После завершения всех воркеров поток сохранения дорабатывает хвост и завершается по сигналу «все воркеры закончили и очередь пуста».
8. Итоговый датафрейм дополняется объединением со всеми собранными объявлениями (на случай хвостов).

---

## 6. Ядро: получение объявлений по родительской категории

### 6.1. Подготовка цели и ветвление по подкатегориям

1. Определяется, сколько объявлений максимум нужно собрать по этой родительской ветке: либо из **подсчёта** на странице категории (фильтр «всё время»), либо при флаге пропуска подсчёта — берётся лимит запроса или запасное число по умолчанию, если лимит нулевой.
2. Эффективный потолок выбора не больше объявленного количества на площадке; минимум одна страница всё равно запрашивается.
3. Загружается страница родительской категории; ожидается появление контента, привязанного к идентификатору категории.
4. Из разметки читается выпадающий список подкатегорий и сопоставляются ссылки; строится список дочерних категорий. Если подкатегорий нет — работа идёт по самой родительской категории.

### 6.2. Цикл по подкатегориям и постраничная выдача

Для каждой подкатегории (или одной «плоской»):

1. Номер страницы начинается с первой.
2. Пока накоплено меньше целевого числа карточек:
   - перед запросом выполняется **адаптивная пауза** (минимальное значение; при проблемах с доступом задержка наращивается, при серии успехов — сбрасывается к минимуму по правилу «чем дольше ждали, тем меньше подряд идущих успехов нужно для сброса»).
   - открывается URL категории с фиксированными опциями сортировки (сначала новые) и номером страницы.
   - ожидается появление блока с данными гидратации Next.js; при таймауте разбор всё равно пробуется, чтобы дать более точную ошибку ниже.
   - из DOM извлекается JSON из тега с идентификатором данных приложения; если в разметке пусто — читается текст того же элемента через движок браузера.
   - если JSON недоступен: проверяются фреймы на признаки ответа «доступ запрещён»; по тексту страницы ищутся признаки rate limit, блокировки, капчи; при пустой или подозрительной странице — ошибка уровня «запрещено» или «элемент не найден»; в отладочном режиме HTML может сохраняться на диск.
3. JSON парсится; из свойств страницы читается код ошибки HTTP, если он передан приложением:
   - **403** — трактуется как rate limit / запрет, с адаптивной логикой и возможным перезапуском браузера или сменой прокси.
   - другие коды — категория считается недоступной, цикл по страницам этой подкатегории прекращается.
4. При успехе фиксируется «успешный запрос» для адаптивной задержки.
5. Из ответа поиска извлекается массив карточек. Если пусто — конец ленты, выход из цикла страниц.
6. Для каждой карточки в порядке выдачи:
   - если уже набран нужный лимит — выход из обхода карточек.
   - пропускаются **рекламные** карточки по префиксу идентификатора.
   - если ID уже в множестве известных:
     - при режиме «лимит равен полному количеству объявлений в категории» уменьшается допустимый потолок (чтобы не крутить ленту бесконечно из-за уже известных сверху);
     - иначе карточка просто пропускается.
   - проверяется соответствие идентификатора дочерней категории ожидаемому для текущей подкатегории, когда активен режим с несколькими подкатегориями (несовпадение — исключение неожиданной категории).
7. Два режима наполнения полей карточки:
   - **Быстрый**: объект объявления строится только из данных выдачи (включая урезанное описание, координаты, признаки срочности и т.д.), без захода на страницу объявления; время публикации в человекочитаемом виде с листинга; просмотры/избранное могут быть нулевыми.
   - **Полный**: с листинга берутся заголовок, ссылки на картинки, локация, вертикали, продавец; затем открывается **страница объявления**: ожидается корневой контейнер; из HTML читаются описание и блоки атрибутов (тип услуги и т.д.); из глобального конфигурационного объекта страницы объявления — тип объявления, цена, статистика, точное время публикации в ISO. При ошибках на странице объявления карточка может быть пропущена с уменьшением целевого потолка. На **403** при открытии детали: сброс счётчика успехов адаптивной задержки; с прокси — ошибка уровня прокси; без прокси — полный перезапуск браузера с теми же параметрами и повтор попытки детали в цикле.
8. Режим **максимального возраста** (например 3 часа): после получения деталей вычисляется возраст объявления относительно текущего времени UTC; если старше порога — выбрасывается специальное исключение с уже накопленным списком (переход к следующей категории). Если время распарсить нельзя — карточка считается пригодной.
9. Если задан коллбек на каждое новое объявление — он вызывается до добавления в общий список.
10. Новые ID добавляются в локальное множество; прогресс-бар обновляется.
11. После обхода страницы, если задан коллбек на пачку — он вызывается со списком карточек с этой страницы.
12. Номер страницы увеличивается; цикл страниц продолжается.

Ошибки внутри страницы: запрет — см. выше; известные ошибки уровня приложения — лог и переход к следующей итерации; таймауты с признаками прокси — перевод в ошибку прокси; потеря сети — пауза и повтор; прерывание с клавиатуры — исключение с частичным результатом; прочие неожиданные — общая ошибка получения списка с возвратом накопленного.

## 7. Режим watch (постоянное слежение)

Отдельный цикл, не смешанный с CLI-пакетным прогоном, но использует тот же класс скрапера:

1. Инициализируется БД; из БД загружаются уже отправленные ID, чтобы не дублировать уведомления.
2. В цикле, пока не запрошена остановка:
   - при отсутствии экземпляра скрапера создаётся новый с укороченными таймаутами, пропуском подсчёта объявлений и включённым режимом, где нужны **реальные временные метки** с карточки детали (не быстрый путь только по выдаче).
   - заново запрашиваются родительские категории;
   - для каждой родительской категории вызывается получение списка с небольшим лимитом на категорию, **без** фильтра по уже известным ID на стороне скрапера (множество пустое), но с порогом возраста в часах;
   - коллбек на новое объявление: если ID ещё не отправляли — добавить в отправленные, upsert в БД, положить в очередь на отправку в Telegram (с сериализацией составных полей).
   - при исключении «устаревшая категория» — тихо перейти к следующей категории.
   - при ошибке прокси — закрыть браузер, обнулить ссылку, подождать фиксированную паузу, повторить цикл.
   - при прочей ошибке — лог, пересоздание браузера при следующей итерации.
3. Между полными проходами по категориям — фиксированная пауза в секундах.
4. При остановке браузер закрывается.

---

## 8. Сводка потоков данных

1. **Каталог**: главная → родительские категории → (опционально) подкатегории.
2. **Листинг**: постранично JSON выдачи → фильтрация рекламы и дубликатов → либо быстрая сборка модели, либо обход карточки детали.
3. **Состояние**: множество известных ID + дедупликация при записи; опционально выкидывание ID для пересканирования по возрасту записи в БД.
4. **Побочные эффекты**: периодический upsert в БД; опционально пакетная отправка в Telegram; в watch — очередь на немедленную отправку новых карточек.
5. **Устойчивость**: очередь прокси, адаптивные задержки, перезапуск браузера при 403 без прокси, разделение ошибок «сменить прокси» и «остановить категорию/весь прогон».

## 9. Прокси (`scraper-golang`)

Отдельная политика для Go-клиента (Playwright и дальнейший скрапер): файл **`proxies.txt`** в корне `scraper-golang` или путь через флаг.

### 9.1. Формат файла

1. Одна строка — один прокси; пустые строки и строки, начинающиеся с `#`, игнорируются.
2. Явная схема:
   - **`socks5://`** или **`socks5h://`** — SOCKS5; при необходимости учётные данные в URL: `socks5://user:pass@host:port`.
   - **`http://`** или **`https://`** — HTTP-прокси (для Playwright сервер задаётся как `http://host:port`; `https://` в файле нормализуется до HTTP CONNECT).
3. Без схемы: строка **`host:port`** считается **HTTP**-прокси (как у Playwright: короткая форма).
4. Если порта нет в URL, подставляются значения по умолчанию: **80** для HTTP, **1080** для SOCKS5.

### 9.2. Когда прокси не используется

1. Файл по умолчанию (`proxies.txt`) не найден ни в текущем каталоге, ни в `scraper-golang/proxies.txt` — браузер идёт **напрямую**.
2. Файл найден, но после удаления комментариев и пустых строк **нет ни одной записи** — **напрямую** (как «прокси в документе нет»).

### 9.3. Проверка перед стартом

1. Если в файле есть хотя бы одна запись, **перед запуском браузера** выполняется проверка доступности прокси (по умолчанию GET через прокси на **probe URL**, например лёгкий HTTPS-хост).
2. Для **HTTP**-прокси используется HTTP-клиент с `Proxy` на этот прокси.
3. Для **SOCKS5** — подключение через SOCKS к целевому `host:443` и TLS-handshake (для HTTPS-probe); для HTTP-probe — простой HTTP-запрос через туннель.
4. Прокси перебираются **по порядку строк**; **первый успешно прошедший проверку** используется для `Launch` Playwright.
5. Если ни один не прошёл проверку — запуск **прерывается** с ошибкой (чтобы не открывать браузер с заведомо мёртвым прокси). Обход: флаг **отключения проверки** (использовать первую строку как есть) — только осознанно.

### 9.4. Интеграция с Playwright

1. Выбранный прокси передаётся в опции **запуска Chromium** (`server` = `socks5://…` или `http://…`, при наличии логина/пароля — отдельные поля учётных данных, как требует Playwright).
2. Дальнейшие навигации и скрапинг идут уже через этот прокси; смена прокси в рантайме = новый `Launch` (для будущего воркера — тот же список и ротация по сбоям).

## 10. Парсинг данных (`scraper-golang`, Playwright)

Реализация в пакете `internal/marktplaats` повторяет цепочку **scraper-python** `MpScraper`: главная → родительские категории (HTML `li.CategoriesBlock-listItem` / запасной `window.__CONFIG__.categoryLinks`) → страница родителя → подкатегории (`select#categoryId` + `a.category-name`) или одна «плоская» категория → постраничный URL `…/p/{n}/#sortBy:SORT_INDEX|sortOrder:DECREASING` → ожидание `#__NEXT_DATA__` → JSON `props.pageProps.searchRequestAndResponse.listings`.

1. **Фильтры**: объявления с `itemId`, начинающимся с `a`, пропускаются (реклама); уже известные `item_id` пропускаются; при нескольких подкатегориях проверяется совпадение `categoryId` строки с ожидаемой подкатегорией.
2. **`errorStatusCode` в pageProps**: `403` трактуется как запрет/rate limit; другие коды — конец ветки категории.
3. **Режим fast** (`-fast`, соответствует `MpScraper(fast=True)`): объект `Listing` собирается из одной строки выдачи (как Python `__listing_from_res_listing`), без захода на карточку.
4. **Режим без fast** (как Python по умолчанию): для каждой строки открывается страница объявления, из HTML читаются описание и атрибуты (`Description-description`, `Attributes-item`), из `window.__CONFIG__` — цена, тип объявления, статистика и поле `since`; опционально сравнение возраста с `max_age_hours` даёт остановку категории с накопленным списком (аналог `CategoryStale`).
5. **Подсчёт объявлений**: при отсутствии `skip_count` загружается страница категории и парсится счётчик у фильтра «Altijd»; при ошибке используется запасная логика как в Python.
6. Точка входа для прогона: `go run ./cmd/marktplaats-playwright/ -scrape` (флаги `-limit`, `-fast`, `-skip-count`, `-parent-index`, `-scrape-base`).
7. В браузере по умолчанию через `page.Route` **не грузятся** типы `image`, `stylesheet`, `font`, а также `media`, `manifest`, `texttrack`, `websocket`, `eventsource`, `ping` (меньше трафика через прокси); **грузятся** `document`, `script`, `xhr`, `fetch`. Отдельных флагов для этого нет — политика `LeanResourcePolicy()` в коде.
